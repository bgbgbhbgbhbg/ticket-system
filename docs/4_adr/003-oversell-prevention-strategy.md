# ADR 003: 防超賣(Oversell Prevention)三層策略

## 狀態
已採用

## 背景
搶票系統最核心的正確性需求:高併發下絕對不能賣超過 `total_quantity`。

## 決策
採用三層防禦,而不是只靠單一機制。

```
Layer 1: Redis 預檢(快速路徑,擋掉大部分「明顯搶不到」的請求)
   ↓
Layer 2: RabbitMQ 序列化(削峰,把瞬間尖峰流量攤平成 worker 可承受的處理速度)
   ↓
Layer 3: DB 樂觀鎖(version 欄位 CAS,最終且唯一真正保證正確性的防線)
```

## 為什麼不能只用其中一層

| 只用哪一層 | 問題 |
|---|---|
| 只靠 Redis 預檢 | Redis 本身在極高並發下也可能有 race condition(INCR/DECR 若沒搭配 Lua script 保證原子性,一樣會超賣);而且 Redis 資料是快取,不是 source of truth,重啟或故障會遺失狀態 |
| 只靠 MQ 序列化 | MQ 只保證訊息依序被消費,不保證消費當下 DB 資料沒有被其他非 MQ 路徑(如未來可能的 admin 手動調整庫存)同時改動 |
| 只靠 DB 樂觀鎖 | 沒有前面兩層擋,DB 會直接承受全部原始流量的樂觀鎖衝突重試,在真正極端流量下(如上萬並發)DB CPU 會被重試邏輯耗盡 |

## 各層的角色定位(重要,常討論)

- **Redis**:效能優化,擋掉「庫存明顯是 0」的請求,減少無效流量進入後面的重量級流程,**不是正確性保證**
- **RabbitMQ**:流量整形(traffic shaping),把突發流量轉成 worker 可控速度處理,降低 DB 瞬間衝突機率,**不是正確性保證**
- **PostgreSQL 樂觀鎖**:唯一的正確性保證,見 `docs/3_specs/data-model.md` 第 2.2 節的 CAS SQL 語法與 `docs/3_specs/domain-state-machine.md` 的重試邏輯

## 後果
- 系統複雜度提高(三層都要維護、都要測試),但正確性風險是搶票系統的第一優先順序,值得這個複雜度成本。
- 若之後要簡化架構(例如被問到「能不能拿掉一層」),答案是:Redis 和 MQ 都可以視流量規模拿掉其中之一甚至兩者都拿掉(直接打 DB),**但 DB 樂觀鎖這層不能拿掉**,因為那是唯一真正防超賣的機制。

## 技術討論重點
「這三層不是疊加起來『三倍安全』,而是分工:前兩層負責效能與流量控制,只有 DB 那層負責正確性。這個分工的好處是,如果之後要簡化架構或縮減成本,我知道哪些是可以拿掉的『優化層』,哪些是不能動的『正確性層』。」
